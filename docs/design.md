# ATLAS Memory 设计思路

> 这是一篇记录设计决策的技术文章，不是功能列表。

---

## 为什么不直接用 RAG？

做这个系统的起点是一个很具体的问题：AI Agent 每次对话都要重新解释背景，用的时间越长，重复成本越高。

朴素 RAG 的典型解决方案是把所有文档向量化，查询时检索 top-k 注入 prompt。这在文档检索场景很合适，但用来做"记住用户说过什么"有两个摩擦点：

**第一，知识有成熟度差异。** 一句随口说的"我今天有点累"和一个经过验证的"用户倾向于在下午三点前完成重要决策"对下次对话的价值完全不同。朴素 RAG 对两者一视同仁。

**第二，时间是天然的相关性信号。** 三年前存下来的偏好可能已经过时。余弦相似度只衡量语义接近，不衡量"这件事现在还重要吗"。

这两个问题导向了同一个结构：记忆需要有层次，检索需要考虑时间。

---

## 四层知识成熟度

系统把记忆分成四个层：

```
L0  原料    用户说的原话、网页里学到的片段、工具返回的数据
L1  知识    经过整理的结构化笔记，有"是什么/为什么/怎么用"
L2  洞见    跨域关联，从多个 L1 中提炼出的模式
L3  框架    稳定的认知模型，指导后续所有决策
```

这个层级不是元数据标签，它影响检索优先级：INJECT 阶段按 L3→L2→L1→L0 的顺序搜索，找到就停。这样高成熟度的知识优先注入，原始记录只在没有更好选择时才出现。

晋升机制是关键：L0 会被 `L1CompletionAgent` 定期扫描，对质量分够高的节点调用 DeepSeek 做结构化整理，产出 L1 文档后写入 Qdrant 和 Obsidian Vault。

## 时间衰减公式的选择

检索得分的最终公式是：

```
final = vector_score × 0.6 + importance × decay × 0.25 + decay × 0.15

decay = exp(-λ × age_in_days)   # λ = 0.05（默认）
```

为什么是指数衰减而不是线性？有两个原因：

1. 遗忘曲线本来就是指数的（艾宾浩斯）。如果记忆的"重要性"和人类记忆模型对齐，注入时机更自然。
2. 指数衰减有个好性质：刚存的记忆衰减慢，但很旧的记忆会快速趋近零——这正是我们想要的行为。

`λ = 0.05` 意味着一条重要性 1.0 的记忆在 14 天后还有 50% 权重，在 30 天后约 22%。可以按领域调整：技术知识衰减慢（λ=0.02），情绪状态衰减快（λ=0.2）。

## 版本链而不是覆盖

最初的实现直接更新已有记忆。这有个隐蔽的问题：如果新信息是错的（AI 幻觉或用户自己说错），旧的正确版本就彻底消失了。

改成版本链后，旧记忆标记 `status: superseded`，新记忆用 `superseded_by` 指向继任者。检索时自动过滤掉 superseded，但它们还在——可以用 `atlas_timeline` 查看一个话题的演变历史，也可以手动回滚。

这个决定稍微增加了存储量，但让系统的可审计性提升很多。

## 反馈回路的设计

`atlas_feedback` 接受三种评价：`correct`（提升分）、`wrong`（降低分）、`outdated`（降低分并建议更新）。负评累积会触发自动压制：feedback_score < 0.5 时不再注入，< 0.2 时删除。

这里有一个取舍：如果用户给了负评但没有补充正确信息，系统会有一个"已知有问题但不知道答案"的空洞。目前的处理方式是把该记忆标记为 superseded 但不立即删除，等待用户补充替代信息。

## L1 课程知识 Pipeline 的设计

课程转录稿→L1 节点这个环节是整个知识库建设中最费力的部分。关键设计决策：

**为什么用 LLM 而不是规则？**  
转录稿是口语，充斥着"就是说"、"对不对"、"那么我们来看"这类填充词。知识点的边界不是固定的——同一段话可能同时包含两个相关概念。规则难以处理这种模糊性，但语言模型在理解语义后再做结构化，天然适合这个任务。

**为什么有 xmind 骨架时效果更好？**  
骨架提供了顶层分类，让模型不需要从零推断这讲"讲了什么"，只需要把内容填入已知框架。对于没有骨架的课程，模型需要同时做"结构推断"和"内容提取"，认知负担更重，容易遗漏细节。

**完整引用块的理由**  
每个知识节点的 `>` 引用块要求完整保留原文，不许摘要。这看起来浪费，但有明确的用途：当你需要核实某个观点的原始出处时，引用块是唯一可信的来源。整理后的书面语段落是解读，原文是证据。

**token 预算计算**  
DeepSeek v4-flash 是推理模型，它在生成 content 之前会产生 reasoning token。实测每次调用约消耗 8000-12000 reasoning token，剩余 budget 才是实际输出。所以 `max_tokens` 的设置是：

```python
target_tokens = min(131072, max(40000, int(len(transcript) * 0.5 / 1.5)) + 12000
```

多加 12000 是为 reasoning overhead 预留的余量。

## 什么没有做好

**benchmark 偏小。** 目前 `retrieval_comparison.py` 只有 10 条记忆和 6 个查询。结论方向是对的（时间衰减重排比朴素向量搜索好），但置信区间很宽。需要至少 100 条标注记忆才能得出统计意义上的结论。

**时间衰减的副作用。** 在 10 条记忆的测试集上，时间衰减重排的 mean P@3 (0.389) 比朴素向量搜索 (0.500) 还低。原因是两条刚存入的记忆（age=0）被公式大幅提升，挤掉了语义更相关但稍旧的结果。"想要最新"和"想要最相关"是两种不同的查询意图，统一用 λ 参数控制太粗糙——理想状态是让用户指定查询意图，或者通过 feedback 数据自动学习权重。

**没有在线学习。** 现在的设计是捕获→存储→检索，但没有机制让系统从"这条记忆有没有帮到用户"的隐式信号中学习。只有显式的 `atlas_feedback` 才能影响分数。

**多用户隔离。** 整个系统是单用户设计，没有命名空间隔离。在多用户场景下需要重新设计。

## 工具链选择的理由

- **Qdrant**：本地部署，REST API 稳定，支持 payload 过滤，不需要 Python 客户端就能操作。
- **bge-m3**：中英文混合效果比 nomic-embed-text 好，1024 维在精度和速度间平衡合理。
- **omlx**：本地推理，避免捕获阶段产生 API 费用。质量评分是过滤低质量记忆的关键门控。
- **DeepSeek v4-flash**：1M 上下文能处理整门课的转录稿，384K 输出不需要分块，价格合理。
